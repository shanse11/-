# 跨层窗口定义与对象映射规范 v0.1

## 1. 研究对象与时间口径

本规范针对 Arbitrum One 的 L1 delayed message / submit retryable transaction。所有 L1 事件采用 Ethereum `finalized` block，记录 block number、hash 与 timestamp；所有 L2 对象记录 receipt 所在 block 的 number、hash 与 timestamp。单位均为秒，差值保留正负号。

| 名称 | 公式 | 时间来源 | 适用对象 | 解释边界 |
|---|---|---|---|---|
| `W_ingest` | `t(L2 retryable creation block) - t(L1 message block)` | L1 `InboxMessageDelivered` 所在 finalized block；由官方 SDK 同构规则派生的 retryable creation receipt 所在 L2 block | 精确映射的 kind=9 submit retryable message | 操作性入块—创建间隔；不是 mempool first-seen 间隔。 |
| `W_effect` | `t(L2 successful redeem block) - t(L1 message block)` | 同上；仅成功 auto/manual redeem receipt 的 L2 block | 已有成功 redeem receipt 的 retryable | 创建、自动 redeem、人工 redeem 分开记录；没有成功 receipt 时为缺失，不填 0。 |
| `P_lag` | `t(L1 batch inclusion) - t(L2 batch last block)` | `SequencerBatchDelivered` 所在 finalized L1 block；NodeInterface 映射区间的 last L2 block | Sequencer batch | L2→L1 publication lag 辅助指标，绝不是 L1→L2 主窗口。 |

## 2. 为什么 batch lag 不是 L1→L2 主窗口

Nitro 的 SequencerInbox 事件在 L1 发布已经排序的数据批次；它反映的是 L2 已产生区块/交易被提交到 L1 的时间关系。L1 用户消息则通过 Inbox/Bridge 的 delayed message 进入，才对应“L1 消息到 L2 创建/执行”的对象链。本项目因而把二者拆开统计。

官方依据：Nitro `ISequencerInbox.SequencerBatchDelivered` 事件与 `NodeInterface.findBatchContainingBlock`；`IBridge.MessageDelivered` 与 `AbsInbox.InboxMessageDelivered`。源码链接：<https://github.com/OffchainLabs/nitro-contracts>。

## 3. finalized 与时间戳限制

- L1 使用 `finalized` tag；run manifest 固定该 anchor 的 hash，避免事后重组歧义。
- L2 creation/redeem 以 receipt 和 block hash 定位；不把 L2 soft confirmation 当作统一统计锚点。
- 历史 RPC 不记录 L1 交易首次进入 public mempool 的时刻，也不记录 Sequencer 私有队列到达时刻；因此不能将 `W_ingest` 宣称为“所有参与者的真实反应窗口”。
- 任何负值、极端值、missing receipt 都进入异常审计；禁止截断为 0 或静默删除。

## 4. retryable 生命周期区分

```text
L1 message 入块
  → L2 retryable creation transaction
  → {auto redeem attempt | 无 auto redeem event | 后续 manual redeem | creation failed}
  → {成功执行 | failed | 尚未观测到成功执行}
```

`W_ingest` 只需要 creation；`W_effect` 只接受成功 redeem。creation failed、auto redeem failed、无 auto event、未发现手工 redeem 分别保存，不能混合。
