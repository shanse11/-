# 三角色威胁模型 v0.1

## 研究边界

场景限定为 Arbitrum One 单 Sequencer、公开 L1 message、普通 L1/L2 提交通道和 finalized 历史数据。讨论的是排序前可见性可能带来的暴露面，不假定已经发生攻击。

| 维度 | Sequencer | 外部策略者 / searcher | 普通用户 |
|---|---|---|---|
| 可见信息 | 公共 L1 消息、直接 L2 提交；可见范围以协议和实际部署为准 | 公共 L1 block/event；若无 live collector，不知道 first-seen | 自己的提交及公共链上信息 |
| 提交渠道 | 维护 L2 排序与 batch 发布路径 | 标准 L1 Inbox 与 L2 RPC | 标准桥和 RPC |
| 排序控制 | 协议边界内的纳入、顺序和延迟选择 | 无直接控制，仅能提交竞争交易 | 无排序控制 |
| 资金/执行 | 可运行排序基础设施；自营/共谋必须显式假设 | 需要资金、路由、gas 与自动化执行 | 常规使用资金与调用能力 |
| 可利用时间 | L1 消息可见至确定 L2 位置前的协议相关时段 | 仅在其能观察并成功提交时 | 不假定策略性反应 |
| 不能做 | 不能伪造 finalized L1、让无效执行被接受，不能无限绕过强制纳入 | 不能默认看私有队列、重排他人或获得 Sequencer 权限 | 不能被默认赋予 searcher/Sequencer 能力 |
| 共谋 | 单独的扩展假设 | 只有明确写出时才成立 | 不讨论 |

## “看到窗口”与“能够获利”必须分开

看到窗口只说明存在可定义的时间关系。能够获利至少还需要：目标内容可用、动作能及时进入排序集合、角色拥有资金和执行路径、反事实顺序有效、扣成本后净收益为正。第 1 周只验证对象/时间关系，不验证后四项。
