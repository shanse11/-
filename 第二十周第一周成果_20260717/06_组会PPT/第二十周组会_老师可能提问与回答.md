# 第二十周组会可能提问与回答（第 1 周）

## 1. 为什么消息窗口长达数分钟，能不能直接说有 MEV？

不能。当前能证明的是：在本次样本中，L1 投递事件与 L2 首次出现 retryable ticket 之间存在可量化的时间差。是否可利用还需要明确攻击者在该段时间是否看到足够信息、能否影响相关状态，以及是否存在可重复的反事实收益；这些均未在本周测量。

## 2. 你如何确认 L1 与 L2 消息不是“看起来相似”的匹配？

匹配采用 retryable ticket 的确定性 ID：从 L1 事件中的 message number、L1 inbox 地址和 L2 chain id 推导，再与 L2 `TicketCreated` 日志的 indexed ticket id 精确比较。原始事件、推导输入输出和 L2 log 均已保存；25 个候选均精确匹配。

## 3. batch 为什么只报告范围，而没有逐笔 L2 交易对应？

SequencerBatchDelivered 对应的是 L2 区块区间；这正是本周能被官方接口稳定复现的粒度。L1 batch 与单笔 L2 交易本来就未必一一对应，强行做逐笔对应会引入不可审计假设。下一周若研究交易位置，另行定义 L2 内部位置变量，不把它伪装成 L1 batch 映射。

## 4. W_effect 为何和 W_ingest 完全一致？

25 个本次样本都在 ticket 创建同一 L2 区块出现 `RedeemScheduled`/`RedeemScheduled` 成功状态，因此以“首个成功 redeem”定义的 effect 锚点与可见锚点重合。这是该预实验样本的观测，不等于协议始终自动执行，也不代表没有失败或延迟 redeem 的路径。

## 5. 为什么不用 Timeboost 指标直接判断？

Timeboost 的拍卖、表达性和延迟优势指标描述特定排序服务机制，不能自动变成跨层消息或 batch 的 MEV 指标。本项目当前以对象可映射、时间锚点可复核为前提；若后续引入 Timeboost，只把其作为另一个机制/对照变量，而非替代跨层可利用性证据。

## 6. 是否已经具备正式实验条件？

目前只满足“一个锚点、小样本、核心字段完整、映射可复现”的初步条件，尚未满足跨时间段稳定性、行为变量可得性和足够样本量。因此下周的第一个产出是准入判定，而不是预设“已经进入正式实验”。

## 7. commit-reveal 是否是最终方案？

不是。本周只把它列为待验证的最小约束方向：在排序前隐藏用户内容，在揭示时验证承诺，以减少排序者提前看到内容的空间。它会带来延迟、失败处理与可用性问题；没有安全证明、Gas 测量或实证防御效果前，不会把它称作方案结论。
