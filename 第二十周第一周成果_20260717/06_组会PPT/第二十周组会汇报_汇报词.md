# 第二十周组会汇报词（第 1 周，2026-07-17）

## 第 1 页｜标题

本周的目标不是直接证明跨层 MEV 已经发生，而是先把“能否观测、能否复现”的证据链搭起来。完成的是定义压实和真实 Arbitrum 数据链路预实验。

## 第 2 页｜本周总览

上周列出的四项工作中，前三项已经落实为可检查产出：窗口与威胁模型写成规范，Arbitrum 小样本链路跑通，阶段一指标形成初版。协议约束仍未启动实现，因为目前还没有行为或利润层面的证据来支撑防御效果判断。

## 第 3 页｜阅读如何服务问题

本周精读的三篇文献分别约束了三件事：BEV 的可识别性不能只靠“看起来异常”；MEV 讨论必须先说明排序者、搜索者与状态边界；收益比较需要反事实而非仅有时间差。它们帮助我们把“窗口存在”与“可利用 MEV”严格分开。

## 第 4 页｜窗口定义

我把指标拆成三个不同对象。对 L1 进入 L2 的 retryable 消息，W_ingest 是 L2 首次可见时刻减去 L1 投递时刻；W_effect 是产生可观察执行结果的 L2 时刻减去投递时刻；对 batch，P_lag 是 L1 发布时刻减去该 batch 最后一 L2 区块的时间。三者不能混用，也不等于攻击时间或盈利机会。

## 第 5 页｜映射与能力边界

数据关系是 L1 Inbox 消息到 Bridge 消息，再到 retryable ticket 和 L2 交易/区块；batch 则通过 NodeInterface 的区间查询得到 L2 区块范围。单 Sequencer 能决定或影响纳入与排序，但普通用户只能提交，外部攻击者只能在公开或可推断的信息到达后行动。本周没有把 Sequencer 的内部私有视图当成可观测数据。

## 第 6 页｜预实验方法

抓取固定在一个 finalized L1 锚点附近的 1,500 个 L1 区块。batch 通过 SequencerBatchDelivered 和 findBatchContainingBlock 映射；消息通过 InboxMessageDelivered、Bridge.MessageDelivered，以及 Arbitrum SDK 同构的 retryable ID 推导精确匹配。全部原始 JSON-RPC 响应、脚本版本、合约代码哈希均已保存。

## 第 7 页｜batch 侧结果

在 40 个连续 batch 样本中，范围映射成功 40 个。P_lag 的范围为 7 到 39 秒，中位数为 16.5 秒，均值为 17.275 秒。这里的事实仅是 L1 发布相对 batch 末端 L2 区块的时间差；不表示这 40 个 batch 存在排序偏置或任何 MEV 收益。

## 第 8 页｜消息侧结果

25 个 retryable 样本均完成从 L1 事件到 L2 ticket 的精确映射，核心字段完整率为 100%。W_ingest 为 404 到 751 秒，中位数 498 秒，均值 551.32 秒。本组样本都在创建 ticket 的同一 L2 区块出现自动 redeem，因此 W_effect 与 W_ingest 相同；这是一种样本状态，不能推广为一般规律。

## 第 9 页｜负结果与边界

失败诊断表没有记录到映射失败，但这只说明本次小样本、当前链路的工程闭环成功。尚未识别交易位置偏置、攻击地址、可疑交易频率、利润、revert 质量或因果效应；因此证据等级停留在可观测的时间与映射事实，不能宣称可利用 MEV。

## 第 10 页｜下一周的判定式推进

下周先扩展多个 L1 时间段，并检查锚点一致性、字段完整性与映射成功率。只有满足准入条件后，才进入正式的描述性实验；否则优先修复链路或收缩到“可观测窗口”的研究问题。Blind commit-reveal 只保留最小机制草案：隐藏排序前内容、在揭示后验证承诺；不声称安全证明、Gas 数据或防御效果。
