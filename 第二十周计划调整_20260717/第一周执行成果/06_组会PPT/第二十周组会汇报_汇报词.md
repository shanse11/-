# 第二十周组会汇报词（约 6–8 分钟）

## 第 1 页｜封面

各位老师好，本周我按照上周计划，没有提前做链上实验或协议编码，而是把研究对象、问题边界和后续验证路线压实。本次汇报的主题是单 Sequencer L2 Rollup 中跨层对象的可见性与排序操纵面。

## 第 2 页｜本周工作严格对齐上周计划

上周要求我继续完善问题定义和系统假设，细化研究问题、威胁模型和预期贡献，并考虑后续实现与验证路线。本周对应完成了这三项：题目从宽泛的跨层 MEV 防御收缩为可见性与排序操纵面；形成了 RQ、角色模型和证据门槛；最后设计了 V0 到 V4 的验证路线。这里的关键纠偏是：batch 向 L1 发布的滞后不能再被当作 L1 到 L2 用户消息的主窗口。

## 第 3 页｜研究对象必须先分清

本项目的主对象是从 L1 发起、之后在 L2 创建或执行的 message/transaction object。L2 直接交易只作对照，而 L2 batch 向 L1 发布只解释 L2 到 L1 的 publication 关系。历史数据可以验证已上链的事件和 receipt，但不能恢复 private Sequencer 队列或真实 first-seen。因此，时间差即使后续可测，也不等于已经发现可利用 MEV。

## 第 4 页｜三角色威胁模型

我把角色明确分开。Sequencer 在协议允许范围内可能影响纳入、延迟和相对顺序，但并不默认恶意、共谋或自营。外部 searcher 默认只能在公开信息出现后提交自己的交易，不能看私有队列或重排他人。普通用户只能提交自己的对象。下一步即使观察到可见性，也还要证明前置状态、有效反事实顺序和扣成本的净收益。

## 第 5 页｜贡献与机制的定位

当前预期贡献首先是概念和模型：把 message visibility 与 batch publication lag 分开，并把能力赋给正确角色。方法上采用先 V0 语义、再 V1 链路、最后才进入收益和机制评估的路线。Commit-Reveal 现在只是一条最小设计方向：它试图隐藏排序前完整 payload，但仍会面对元数据泄漏、超时、审查和强制纳入问题，因此不能把它讲成已安全或已有效的方案。

## 第 6 页｜下周推进方式

下周优先做 V0 和 V1。V0 核对官方对象语义、时间锚点和角色边界；V1 才准备小样本数据链路，且只接受唯一 ID 或官方事件关联，不以时间最近补配。若对象无法稳定映射，就收缩研究问题，而不是通过不可靠数据硬做窗口和收益结论。
