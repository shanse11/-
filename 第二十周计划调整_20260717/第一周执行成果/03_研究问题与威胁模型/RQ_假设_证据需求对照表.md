# RQ—假设—证据需求对照表

## RQ1：跨层对象在不同阶段对哪些角色可见？

| 项目 | 定义 |
|---|---|
| 精确定义 | 对一个 L1 发起、在 L2 创建或执行的对象，分别列出 L1 构造/入块、排序前、L2 创建、L2 执行阶段中 Sequencer、外部 searcher 和用户可获得的信息。 |
| 当前假设 | Sequencer 对其已接收且待排序对象可能拥有比外部观察者更早或更完整的信息；具体实现和范围不得预设为事实。 |
| 可观察量 | L1/L2 block、receipt、event、对象唯一标识/关联字段、提交接口公开语义。 |
| 所需证据 | V0 官方文档/合约/接口核对；V1 对象映射的原始响应。 |
| 反例或证伪 | 某对象类型没有唯一关联、没有在 L2 创建，或公开接口不支持“进入 Sequencer 前”的语义，则不能将其纳入该对象模型。 |
| 后续验证方法 | 逐类对象建立生命周期表；将可验证链上阶段与不可验证私有阶段分栏。 |
| 本周可回答 | 可见性必须按角色、阶段、证据类型拆分；历史 RPC 不能证明私有 first-seen。 |
| 本周不能回答 | 任一具体对象实际何时被某 Sequencer 首次看到，或该事实是否早于公开观察者。 |

## RQ2：把可见信息转化为收益，需要哪些能力与状态条件？

| 项目 | 定义 |
|---|---|
| 精确定义 | 给定对象信息和角色，确定其是否同时具备合法动作、可重建前置状态、有效反事实顺序和扣除成本后的正净收益。 |
| 当前假设 | Sequencer 的纳入/相对排序能力与外部 searcher 的“只提交自己交易”能力不同；普通用户不自动具有策略性操作能力。 |
| 可观察量 | receipt/status、交易位置、前置状态、gas/费用、资产变动、可重复执行条件。 |
| 所需证据 | V2 描述性数据；V3 预注册候选规则、状态快照/重放、统一计价和成本口径。 |
| 反例或证伪 | 反事实顺序不可执行、收益不足以覆盖成本、角色没有所需排序权限，均使“可利用性”不成立。 |
| 后续验证方法 | 使用《最小可利用性见证模板》逐个候选填证据；缺一项即保留为未确认。 |
| 本周可回答 | 可利用性是多条件链，不是时间差或位置靠前的同义词。 |
| 本周不能回答 | 任一现实主体已获利、攻击频率、实体身份、因果关系。 |

## RQ3：最小可见性约束应限制什么、保留什么并承担什么代价？

| 项目 | 定义 |
|---|---|
| 精确定义 | 为受限对象确定排序前需隐藏的内容、允许公开的元数据、揭示/验证阶段、超时处理及与跨层生命周期的兼容要求。 |
| 当前假设 | `commit = H(payload || salt || domain)` 可以作为机制草案起点；承诺本身不自动隐藏低熵 payload 或元数据。 |
| 可观察量 | 协议接口、commit/reveal 状态、超时状态、延迟、失败/拒绝服务条件、兼容性约束。 |
| 所需证据 | V4 威胁—性质映射、实现原型、性能/成本测量与安全分析。 |
| 反例或证伪 | payload 可被字典猜测、元数据足以泄漏策略、未 reveal 造成不可接受的阻塞，均缩小机制主张。 |
| 后续验证方法 | 定义 hiding/binding/活性/抗审查等待验证性质，并对比无约束基线。 |
| 本周可回答 | 机制应服务于“排序前内容可见性”这一目标，并列出风险。 |
| 本周不能回答 | 已安全、已部署、Gas 可接受、已降低 MEV。 |
