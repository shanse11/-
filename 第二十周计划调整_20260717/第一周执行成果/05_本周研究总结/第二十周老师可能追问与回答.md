# 第二十周老师可能追问与回答

## 为什么不再以“窗口多长”为核心问题？

时间差只是一类后续可观察量。若不先说明对象、可见主体、行动权限、状态和成本，窗口长度无法证明可利用性。当前核心问题改为“可见性在何种能力条件下形成操纵面”。

## 为什么 batch publication lag 不是主研究对象？

batch 向 L1 发布描述的是已形成 L2 批次的 L2→L1 关系；主对象是 L1 发起、在 L2 创建或执行的 message。对象和方向不同，不能由前者反推后者的用户窗口。

## 怎样定义 Sequencer 的可见性？

本周把它拆为公共链上可验证信息与排序前私有信息。公共部分可由 block/receipt/event 核验；私有队列的真实 first-seen 和内容范围是系统假设，必须在 V0 通过具体接口和架构资料收紧。

## 为什么不能默认 Sequencer 恶意？

研究模型应先给出能力，再单独加入行为或共谋假设。排序权本身不是作恶证据；若预设恶意，会把待验证结论写进前提。

## 外部 searcher 和 Sequencer 有何不同？

searcher 默认只能基于公开信息提交自己的交易；Sequencer 对自身排序集合具有纳入、延迟和相对排序影响。不能把后者的重排能力自动赋给前者。

## 可见性如何才会转化为 MEV？

还需要合法行动能力、可重建前置状态、有效的反事实顺序和扣除 gas/费用后的正净收益。任何一项缺失，最多是风险候选而非确认 MEV。

## commit-reveal 能解决什么、不能解决什么？

它可能减少排序前对完整 payload 的直接可见性，并用 commitment 验证揭示内容；但仍可能泄漏元数据，且不自动解决审查、强制纳入、reveal 后重排、低熵猜测或性能代价。

## 为什么本周没有做实验？

本周任务就是理论与方案定义。先确定对象语义、字段和失败分类，才能避免把不可靠映射包装成窗口或 MEV 结论。实验性产出已体现为 V0–V4 路线和 V1 设计。

## 下一步如何验证？

先 V0 核对官方对象语义与唯一关联规则；再 V1 对少量 eligible message 验证数据链路。只有映射稳定、字段完整、失败可解释后，才进入描述性指标或更高层验证。

## 如果真实对象无法映射怎么办？

保留 `unmapped` 和失败原因，不以时间最近补配；将研究问题收缩到“可见性与角色能力模型”，或选择具有官方唯一关联的更窄对象类型。
