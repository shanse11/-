# 第二十周第 1 周科研计划（修订版）

> 对齐依据：2026-07-10 组会中明确的下周计划——（1）继续完善题目对应的问题定义与系统假设；（2）进一步细化研究问题、威胁模型和预期贡献；（3）考虑后续实现与验证路线。
>
> 本周定位：**理论与方案定义周**。不把数据抓取、窗口统计或协议实现当作本周必须完成的实验任务；实验只形成可执行的验证设计和准入条件。

## 0. 范围校正

本周不回答“Arbitrum 上是否已存在可利用跨层 MEV”，也不承诺实现 Blind Commit-Reveal。应先把研究问题写成一个可证伪、可验证的问题：在单 Sequencer 的 Rollup 中，哪些跨层对象会在排序前暴露给谁；这种暴露在何种能力假设下才可能转化为排序操纵或收益机会；若确有风险，最小可行的可见性约束应满足什么性质。

## 1. P0：问题定义与系统假设（预计 1.5–2 天）

### 1.1 建议题目与主问题

暂定题目：**单 Sequencer L2 Rollup 中跨层交易可见性与排序操纵面：问题定义、威胁模型与最小约束设计**。

主问题不是“窗口有多长”，而是：

> 对于从 L1 进入 L2、且会在 L2 触发可执行动作的跨层对象，单 Sequencer 在排序前获得的内容、到达顺序或关联状态信息，是否构成一个需要被约束的操纵面？若构成，该约束应作用于哪些信息与阶段？

### 1.2 系统假设（写入问题定义）

| 维度 | 本周采用的假设 | 不应额外假设 |
|---|---|---|
| 链结构 | Ethereum L1 + Arbitrum 类 optimistic Rollup；L2 由单一 Sequencer 运营 | 不假设所有 Rollup 的消息生命周期完全一致 |
| 研究对象 | L1 发起、在 L2 创建或执行的消息/交易对象；L2 直接交易仅作对照 | 不把 L1 batch 发布滞后当作 L1→L2 用户主窗口 |
| 排序阶段 | Sequencer 对进入其排序集合的对象具有协议允许范围内的纳入、延迟和相对顺序影响力 | 不预设其必然恶意、必然自营交易或与搜索者共谋 |
| 可观测性 | 历史链上数据可验证入块、事件、receipt 与最终状态 | 不把历史 RPC 当作 mempool first-seen 或 Sequencer 私有队列记录 |
| 目标 | 识别“信息可见性—行动能力—潜在操纵”的条件链 | 不直接声称已观测 MEV、利润或攻击频率 |

### 1.3 可交付与判定标准

- 交付：一份 2–3 页的问题定义与系统模型，包含对象边界、时间阶段、术语表和反例。
- 判定：能清楚回答“何种对象在何时进入 Sequencer 可见范围”“batch lag 为什么不是主问题”“为什么窗口本身不足以构成 MEV 结论”。

## 2. P0：研究问题、威胁模型与预期贡献（预计 2 天）

### 2.1 三个递进研究问题

| 编号 | 研究问题 | 本周应给出的答案形式 | 不能提前给出的答案 |
|---|---|---|---|
| RQ1：可见性 | 跨层对象在 L1 入块、L2 创建和 L2 执行之间，哪些内容对哪些角色可见？ | 分阶段信息集与对象关系图 | “存在 N 秒可利用窗口”的实证结论 |
| RQ2：可利用性 | Sequencer 或外部策略者把可见信息转为收益，必须同时具备哪些行动能力与状态条件？ | 必要条件清单、反事实收益模板 | “已经发生排序操纵” |
| RQ3：约束设计 | 若要降低排序前内容可见性，最小机制应限制什么、保留什么功能、承担什么代价？ | Blind commit-reveal 的目标、接口草案、待验证性质 | 安全证明、Gas 数字、部署可行性或防御效果 |

### 2.2 威胁模型（以能力而非结论组织）

| 角色 | 可见信息 | 可行动作 | 本周允许讨论 | 本周不讨论 / 不默认成立 |
|---|---|---|---|---|
| Sequencer | 其收到并准备排序的对象；公共 L1/L2 状态 | 纳入、延后、相对排序；是否插入自营交易需另设假设 | “若提前见到内容且有排序权，可能形成操纵面” | 恶意、共谋、自营、私有队列具体内容 |
| 外部 searcher | 公共事件、公共状态；可能通过标准入口提交交易 | 自己的交易提交与执行 | 在公开信息到达后能否竞争 | 看见私有队列、重排他人、获得 Sequencer 权限 |
| 普通用户 | 自己的提交与公共状态 | 发起跨层消息或 L2 交易 | 被动暴露和执行质量风险 | 具有搜索、重排或审查能力 |

最小威胁命题：**若 Sequencer 在承诺顺序前可读取会影响 L2 状态的对象内容，且其能够调整相关对象的相对位置，则需要评估是否存在可执行、可获利的反事实顺序。**

这里的“需要评估”不是“已经成立”。收益仍须满足：可观察信息、允许动作、前置状态、有效执行、成本扣除和反事实比较。

### 2.3 预期贡献（可写入开题/论文，但使用条件语气）

1. **概念贡献**：区分 L1→L2 消息可见性阶段与 L2→L1 batch publication lag，避免以单一“跨层窗口”混淆两类对象。
2. **模型贡献**：给出单 Sequencer、外部策略者、普通用户的分层威胁模型，并明确从可见性到可利用性所缺失的条件。
3. **方法贡献**：提出一套先做对象语义与可复核映射、再做候选行为与反事实收益的验证路线。
4. **设计贡献（待验证）**：提出只在排序前隐藏必要内容、揭示后验证承诺的最小机制方向；不在本周承诺完整协议。

### 2.4 可交付与判定标准

- 交付：RQ—假设—可观察量—证据要求对照表；三角色能力矩阵；一页“可利用性见证”模板。
- 判定：每个预期贡献都能对应一个研究问题和后续验证动作；任何“可利用 MEV”措辞都能被拆回尚缺的证据条件。

## 3. P1：后续实现与验证路线设计（预计 1–1.5 天）

### 3.1 验证路线：先语义，后链路，最后行为

| 阶段 | 目的 | 本周产出 | 进入下一阶段的准入条件 |
|---|---|---|---|
| V0：对象语义核对 | 确认 L1 message、L2 创建/执行、batch 的官方含义与可对应粒度 | 事件/接口/字段清单；时间锚点定义 | 对象关系不依赖“时间最近”配对；batch 与 message 不混用 |
| V1：小样本数据链路验证 | 验证所需字段能否从真实链上得到并可复现映射 | 采集脚本设计、golden-set 规则、失败分类表 | 明确唯一标识或官方映射规则；原始响应可保存 |
| V2：描述性实验 | 描述窗口/映射质量等可观测事实 | 指标表、样本策略、图表模板 | 多时间段稳定、字段完整、映射成功率达预设门槛 |
| V3：行为与收益验证 | 评估候选操纵及反事实收益 | 预注册启发式、状态重放与成本口径 | 有前置状态、允许动作、反事实序列、净收益定义 |
| V4：机制评估 | 比较最小约束的安全性与代价 | 安全性质清单、接口和基准方案 | 威胁目标明确；实现与性能测试环境可获得 |

### 3.2 本周只制定、不执行的最小数据链路方案

- 对象优先级：先选具有唯一 ID 派生或官方事件关联的 L1→L2 retryable/delayed message；不以时间邻近补配。
- 最小字段：L1 block/hash/log index、message number/kind、L2 creation/execute receipt、L2 block/hash/timestamp、mapping status/method/failure reason。
- 小样本规模建议：20–30 条 eligible message 和 30–50 个 batch，仅用于验证字段与映射；batch 只辅助说明 L2→L1 发布关系。
- 失败分类预置：RPC 缺历史、事件不完整、对象不适用、ID 派生失败、L2 receipt 缺失、生命周期未终结、时间戳负值。
- 预注册边界：不把窗口统计用作攻击频率、排序偏置、地址归属、MEV 利润或防御效果。

### 3.3 本周只制定的最小机制草案

`commit = H(payload || salt || domain)`；排序前提交 commitment，揭示阶段提供 payload 与 salt，合约/排序层验证 commitment 后再执行或纳入。

本周只讨论四项：

1. 目标：减少排序前对完整内容的直接可见性；
2. 需保留：用户可提交、承诺可验证、超时/未揭示可处理；
3. 待验证：延迟、拒绝服务、信息泄漏、强制纳入、跨层消息生命周期兼容性；
4. 不得声称：已证明安全、已测 Gas、已部署或已降低 MEV。

### 3.4 可交付与判定标准

- 交付：一张验证路线图、一张字段/映射设计表、一张最小机制目标—性质—风险表。
- 判定：下一步任何工程工作都能说明“验证哪一个 RQ、需要哪一类证据、失败时如何收缩问题”。
